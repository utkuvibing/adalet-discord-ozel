**4. Update `src/server/middleware/auth.ts` — Dual auth path (session token OR invite token)**

Refactor the auth middleware to support TWO authentication flows:

**Flow A — Returning user (has sessionToken):**
- Check `socket.handshake.auth.sessionToken`
- If present, call `findUserBySession(sessionToken)`
- If valid user found: set `socket.data.userId`, `socket.data.displayName`, `socket.data.avatarId`, `socket.data.sessionToken` and call `next()`
- If invalid session: call `next(new Error('INVALID_SESSION'))` — let client fall back to join form

**Flow B — New user (has invite token) — existing logic, enhanced:**
- Existing invite token validation stays as-is
- After validating invite token: call `createUser(displayName, avatarId)` to persist user in DB
- Set `socket.data.userId`, `socket.data.displayName`, `socket.data.avatarId`, `socket.data.sessionToken`
- The server must emit the session token BACK to the client (via a new `session:created` event — add to ServerToClientEvents) so the client can save it to localStorage

**Flow C — Host localhost bypass:**
- Modify existing localhost bypass: check if host has a saved session first (via sessionToken in auth)
- If host has valid session: use it (same as Flow A)
- If no session: create a new user with `displayName` from auth (defaulting to 'Host') and `avatarId` from auth (defaulting to first preset)
- Set socket.data fields and continue

In the `registerSignalingHandlers` function or right after auth in the connection handler, emit `session:created` for new users (Flow B and Flow C first-time). For returning users (Flow A), emit `session:created` with their existing data so the client can refresh localStorage.
