```typescript
let currentPTTAccelerator: string | null = null;

ipcMain.handle('ptt:register', (_event: Electron.IpcMainInvokeEvent, accelerator: string) => {
  // Unregister previous if exists
  if (currentPTTAccelerator) {
    globalShortcut.unregister(currentPTTAccelerator);
  }

  try {
    // Electron globalShortcut doesn't support keyup natively.
    // Use a polling approach: register shortcut, send pressed=true,
    // then use a short interval to detect release.
    // Alternative: use 'keydown' and 'keyup' via different mechanism.
    // For Electron, use globalShortcut for keydown detection.
    const registered = globalShortcut.register(accelerator, () => {
      mainWindow?.webContents.send('ptt:state-change', true);
      // Set a timeout to send release -- globalShortcut fires repeatedly while held
      // The renderer will handle debouncing
    });
    if (registered) {
      currentPTTAccelerator = accelerator;
    }
    return registered;
  } catch {
    return false;
  }
});

ipcMain.on('ptt:unregister', () => {
  if (currentPTTAccelerator) {
    globalShortcut.unregister(currentPTTAccelerator);
    currentPTTAccelerator = null;
  }
});
```

**IMPORTANT NOTE on PTT keyup detection:** Electron's `globalShortcut` does NOT fire on keyup. The recommended approach for hold-to-talk is:
- `globalShortcut.register` fires on keydown (sends `ptt:state-change` true)
- Use a `setInterval` (every 50ms) that checks if the key is still held. Since globalShortcut fires repeatedly while held, track the last fire time. If no fire for >100ms, the key was released (send `ptt:state-change` false).
- Alternatively, use the `uIOhook` npm package for proper keyup. But to avoid new dependencies, use the repeat-detection approach:
  - On each globalShortcut fire: set `lastPTTFire = Date.now()` and send `pressed=true` if not already pressed
  - A 100ms interval checks: if `Date.now() - lastPTTFire > 150` and currently pressed, send `pressed=false`

Implement this repeat-detection approach in main.ts. Store `pttPressed` and `lastPTTFire` state variables in the registerIpcHandlers scope. Start the interval when PTT is registered, clear it when unregistered.

**Core implementation:**

a) **getUserMedia on room join:** When `activeRoomId` changes from null to a number, call `navigator.mediaDevices.getUserMedia({ audio: { echoCancellation: true, noiseSuppression: true, autoGainControl: true } })`. Store the MediaStream in a ref. When `activeRoomId` becomes null, stop all tracks.

b) **Add audio track to peer connections:** After getting the local stream, iterate `peerConnections.current` and call `pc.addTrack(audioTrack, localStream)` for each peer connection. This triggers renegotiation via onnegotiationneeded (already handled by useWebRTC).

c) **Handle new peers joining AFTER audio is active:** Use a `MutationObserver`-style approach: export a callback `onPeerAdded` or watch the peerConnections map. Simpler: in useWebRTC, add an `onTrackCallback` ref that useAudio sets. When a new peer is added, useWebRTC calls the callback with the new RTCPeerConnection so useAudio can add its track.

Actually, the cleaner approach: **Modify useWebRTC to accept an optional `localStream` ref.** When `addPeer` creates a new RTCPeerConnection, if `localStream.current` exists, add all audio tracks to the pc before creating the data channel. This way tracks are added before negotiation starts. Remove the keepalive data channel creation since audio tracks will trigger negotiation.

d) **Receive remote audio -- ontrack in useWebRTC:** Modify `addPeer` in useWebRTC to set `pc.ontrack = (event) => { onTrackCallback(remoteSocketId, event.streams[0]) }`. The `onTrackCallback` is a ref passed from useAudio.

e) **Web Audio API routing for remote streams:** For each remote stream received via ontrack:
- Create `AudioContext` (singleton, created once)
- Create `MediaStreamSource` from the remote stream
- Create `GainNode` (for per-user volume control, exposed via `setRemoteVolume`)
- Create `AnalyserNode` (for speaking detection, used by Plan 03-02)
- Connect: source -> gainNode -> analyser -> audioContext.destination
- Store in `remoteStreams` Map keyed by socketId

f) **Voice state management:** Track `myVoiceState` as React state `{ muted: false, deafened: false, speaking: false }`. When muted changes, enable/disable the local audio track (`localStream.getAudioTracks()[0].enabled = !muted`). When deafened changes, set all remote GainNodes to 0 (deafened) or restore previous values. Emit `voice:state-change` to socket whenever myVoiceState changes.

g) **Reconnect watchdog:** Listen for `connectionState` changes on each RTCPeerConnection. If state goes to `'failed'`, attempt ICE restart: `pc.restartIce()`. If that fails after 5 seconds, remove and re-add the peer via `removePeer` + `addPeer`. Socket.IO already handles reconnection (configured in useSocket with infinite retries). On socket reconnect event, re-join the active room by emitting `room:join` again -- this triggers fresh `room:peers` and re-establishes WebRTC. Add a useEffect that watches `connectionState` from useSocket -- when it transitions from `'reconnecting'` to `'connected'` and `activeRoomId` is set, re-emit `room:join`.

h) **Listen for remote voice state changes:** Listen on socket for `voice:state-change` events. Store in a `voiceStates` Map<string, VoiceState> keyed by socketId. Update when received. Clean up when peer disconnects.
